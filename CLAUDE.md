# CLAUDE.md

Claude Code が各セッション開始時に読むプロジェクトレベルのルール。

## リポジトリの目的

複数プロジェクトを管理するモノレポ。各プロジェクトは `projects/<project-name>/` 配下に独立配置する。

## 現在のプロジェクト

| プロジェクト | 概要 | 場所 |
|---|---|---|
| **DELNE** | 認知症者向けAI電話サービス。現在はマーケティング活動が主な作業対象 | `projects/delne/` |

新規プロジェクトを追加する場合は `projects/<name>/` を作成し、本テーブルにも追記する。

## コミュニケーション

- **言語**: 日本語 (応答・コミットメッセージ・ドキュメント全て)
- **トーン**: 簡潔・端的。冗長な前置き・装飾・過剰サマリは禁止
- **スコープ**: 1ターン1論点。複数論点を並べず、1件決まったら次へ進む
- **出力量**: 要点のみ。表・コードブロック・引用は必要なときだけ使う
- **次の論点提示**: 1論点が決着したら、次に議論したいテーマを選択肢方式 (2-4択) で提示する。ユーザーが選んでから次の議論に入る
- **平易な言い換え**: Claude Code の専門用語 (セッション・スレッド・コミット・push 等) や内部ファイル名は、ユーザー向け対話では平易な日常語に言い換える (例: 「セッション」→「いま開いているチャット」、「コミット」→「保存」、「`_threads/active.md`」→「スレッドの状況メモ」)。括弧書きの注釈ではなく、最初から日常語で書く
- **例外**: 成果物・ファイル本体・プロンプト本体など「ユーザーが他所で利用する文書」は必要な詳しさを保つ (簡潔化の対象は対話文のみ)

## Git運用

- **作業ブランチ**: `main` のみで運用する。フィーチャーブランチは作らない (単独運用前提)
- **コミット・push の確認**: Claudeは確認なしで直接 main に commit & push してよい
- **Claude Code on the web の自動ブランチ**: 新規セッションで自動的にフィーチャーブランチが作られる場合は、セッション冒頭で `git checkout main` に切り替えてから作業する
- **コミットメッセージ**:
  - 1行目: `<type>: <要約>` (type は chore / feat / fix / docs / refactor / content 等。要約は日本語OK)
  - 本文: 必要に応じて変更理由を日本語で記述
- **例外 (事前確認が必要な操作)**: force push、`git reset --hard`、`git branch -D`、ファイル大量削除、履歴改変など破壊的・不可逆な操作のみ事前確認する

## DELNEプロジェクト固有のルール

### 製品情報の必須参照

DELNE について議論・作業する際は、**必ず以下のファイルを最初に参照** する:

- **`projects/delne/product/summary.md`** — DELNE 製品サマリ (single source of truth)
  - コンセプト・ターゲット・機能・料金・マーケティング方針・実績などを集約
  - 製品情報に関する記述・主張は本ファイルと矛盾しないようにする
  - 新事実が確定したら本ファイルを更新する (本ファイルが常に最新)

### 領域の特性 — 認知症・医療領域

DELNE は認知症の方を対象としたサービス。以下に常に注意:

- **薬機法・医療広告ガイドライン**: 治療効果・改善効果の断定的表現は避ける
- **誇大表現の回避**: 「必ず」「治る」「効く」「医療レベルの〜」等は基本NG
- **配慮表現**: 「認知症患者」より「認知症の方/認知症のある方」など、当事者の尊厳を尊重した表現を選ぶ
- **家族介護者への配慮**: 罪悪感を煽る表現 (「あなたが見ていないと…」等) は避ける

具体的な表現規範は **`projects/delne/marketing/strategy/compliance.md`** を参照すること。全てのコンテンツ作成時はこの規程に照らして自己チェックする。

### 個人情報の取り扱い

- ユーザーインタビューの**録音・生回答・個人特定可能情報はコミットしない**
- 要約・分析結果のみ `projects/delne/marketing/research/` に格納
- 顧客名・電話番号・メールアドレス等を含むファイルは作成しない

## 作業スレッドの分け方 (Claude Code on the web 運用)

**軽量シングル戦略** で運用する (立ち上げ初期向け)。
必要に応じて将来「ペルソナ別企画」「カテゴリ別企画」に拡張する。

### スレッド間の状態共有

並列セッション間の調整は **`_threads/` ディレクトリ** を介して行う。

- `_threads/README.md` — 運用ルール (セッション開始/終了時の手順、コンフリクト回避)
- `_threads/active.md` — 全スレッドの現状サマリ (各スレッドが自セクションを更新)
- `_threads/<thread-name>/brief.md` — 各スレッドのスコープ・参照ファイル
- `_threads/<thread-name>/log.md` — 各スレッドの意思決定・成果物ログ

全スレッドは **セッション開始時に `_threads/active.md` と自スレッドの brief.md・log.md を必ず読む** こと。
セッション終了時には自スレッドの log.md と active.md (自セクションのみ) を更新して push する。

### スレッド構成

| 種別 | 役割 | 本数 | 寿命 |
|---|---|---|---|
| **統合企画スレッド** | 全カテゴリ横断の方針・優先順位・意思決定の場 | **1本** | 長期維持 |
| **作業スレッド** | 個別成果物 (LP1ページ・X投稿1本・プレス1件等) の制作 | 都度 | 短期完結 (作業後は放置) |

### 統合企画スレッド

- スレッド名: `DELNE-marketing 統合企画`
- 役割:
  - 全体方針・優先順位の議論
  - どの個別成果物を作るかを決定
  - カテゴリ横断の意思決定 (LP/SNS/プレス/広告/営業資料/リサーチ等)
  - 戦略・コンプラ・ブランドガイドライン等の全体ルール更新もここで議論
- 1本に集約することで、文脈の分断を防ぎ、1人運用での管理コストを最小化

### 作業スレッド

- スレッド名例: `X投稿: 2026-06-01 ローンチ告知`, `プレス: サービス開始`, `LP: 企業向けセクション追加`
- 役割:
  - 1つの成果物に集中して制作
  - 統合企画スレッドでの判断を踏まえて実施
  - 作業完了後はそのまま放置でOK (再利用しない)

### スレッド切り出しの目安

| 状況 | スレッド種別 |
|---|---|
| 方向性を決めたい / 全体俯瞰したい / カテゴリ横断 | 統合企画スレッド |
| 具体的な成果物を1つ作る | 作業スレッド (都度新規) |
| 既存ルール (CLAUDE.md・compliance.md等) の更新 | 統合企画スレッド |

### 将来の拡張パス

立ち上げ初期を脱して以下が起きたら、スレッド構造を見直す:

1. **ペルソナ別の議論が増えた** → 「家族介護者向け企画」「企業向け企画」「プレゼント送り手向け企画」のように **ペルソナ別企画** に分割
2. **媒体別のノウハウ蓄積が必要になった** → 「LP企画」「SNS企画」「広告企画」のように **カテゴリ別企画** を追加
3. **時期/フェーズで管理したい** → 「2026年Q3企画」のように **タイムボックス企画** を導入

拡張時は本セクションを更新する。
